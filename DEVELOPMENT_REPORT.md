# 주식 포트폴리오 관리 사이트 · 개발 보고서

> 목적: 2개 증권사에 보유한 주식을 한 화면에서 관리하는 웹사이트의 전체 개발 과정과 구조를 기록한다.
> 이후 기능 수정·확장 시 참조하는 것을 목표로 하며, 비개발자도 어디를 고쳐야 하는지 찾을 수 있도록 작성한다.

- **작성일**: 2026-07-15
- **서비스 주소(정식)**: https://kospi20k.vercel.app/
- **상태**: 실시간 시세 조회 정상, 클라우드 저장(여러 기기 동기화) 정상

---

## 1. 요구사항 요약

| 항목 | 내용 |
|---|---|
| 목적 | A·B 두 증권사의 보유 주식을 통합 관리 |
| 입력 항목 | 종목명, 종목코드, 보유수량, 매입평균가 (직접 입력) |
| 현재가 | API로 자동 조회 (실시간 시세) |
| 자동 계산 | 종목별·증권사별·전체 합산의 평가손익 및 수익률 |
| 데이터 저장 | 어느 브라우저·기기에서 접속해도 동일한 데이터 |
| 배포 | 웹사이트로 배포 (누구나 주소로 접속) |

## 2. 최종 아키텍처

```
[사용자 브라우저]
   │  (1) 화면 로드            (2) 현재가 조회         (3) 데이터 저장/불러오기
   ▼                              │                       │
index.html  ────────────────────►  /api/price  ──────►  한국투자증권(KIS) Open API
 (HTML/CSS/JS 단일 파일)           /api/data   ──────►  Upstash Redis (KV 저장소)
   ▲                                (Vercel 서버리스 함수)
   └── localStorage (오프라인 캐시)
```

**핵심 설계 원칙**: 증권사 API 키(시크릿)와 저장소 토큰은 브라우저에 절대 노출하지 않는다.
브라우저는 항상 우리 서버(Vercel 서버리스 함수)를 거쳐서만 외부 API를 호출하며,
비밀 값들은 서버의 환경변수에만 보관한다.

## 3. 기술 스택 및 선택 이유

| 구성 | 선택 | 이유 |
|---|---|---|
| 프런트엔드 | 순수 HTML/CSS/JavaScript 단일 파일 | 빌드 도구·프레임워크 없이 그대로 배포 가능, 유지보수 단순 |
| 백엔드 | Vercel 서버리스 함수 (Node.js) | 별도 서버 운영 불필요, GitHub 연동 시 자동 배포, 무료 |
| 현재가 API | 한국투자증권 KIS Open API | 국내주식 실시간 시세 무료 제공 |
| 저장소 | Upstash Redis (Vercel KV) | 키-값 하나 저장에 적합, Vercel에서 원클릭 연결, 무료 티어 |
| 호스팅 | Vercel + GitHub 자동 배포 | 커밋하면 자동 재배포 |

## 4. 주요 의사결정 로그 (변경 이력)

개발 과정에서 방향이 바뀐 지점과 그 이유를 기록한다. 되돌리거나 재검토할 때 참고한다.

| 시점 | 결정 | 배경 / 이유 |
|---|---|---|
| 초기 | 현재가 API로 **KIS Open API** 채택 | 국내주식 실시간 시세를 무료로 제공. 단, 브라우저 직접 호출은 CORS·시크릿 노출 문제 → 프록시 서버 필요 |
| 중간 | **공공데이터포털 API로 교체 검토** | 사용자가 data.go.kr 키 보유. 계좌·OAuth 불필요해 더 간단 |
| 재검토 | **KIS로 복귀(최종)** | 공공데이터포털은 "일별 종가"라 실시간이 아님(다음 영업일 오후 갱신). 실시간 시세가 필요해 KIS 유지로 결정 |
| 저장 | localStorage → **KV(클라우드) 저장 추가** | localStorage는 브라우저마다 별도 저장이라 다른 기기에서 초기화됨. 기기 간 동기화 위해 서버 저장 도입 |

> 참고: 공공데이터포털 방식은 실시간이 아니라는 제약으로 최종 채택하지 않았다.
> 만약 향후 "일별 종가로 충분, 계좌 없이 운영"으로 방침이 바뀌면 9.4절 참고.

## 6. 화면(index.html) 상세

### 6.1 데이터 모델

브라우저 메모리와 저장소에 공통으로 쓰이는 구조는 다음과 같다.

```js
{
  brokers: [
    {
      name: "A증권",
      rows: [
        { name:"삼성전자", code:"005930", qty:10, buy:70000,
          price:75000, manual:false, err:null }
      ]
    },
    { name: "B증권", rows: [ ... ] }
  ]
}
```

| 필드 | 의미 | 비고 |
|---|---|---|
| `name`(broker) | 증권사 이름 | 화면 상단에서 직접 수정 가능 |
| `rows` | 보유 종목 목록 | |
| `name`(row) | 종목명 | 사용자 입력값 사용 |
| `code` | 종목코드(6자리) | 현재가 조회 키 |
| `qty` | 보유수량 | |
| `buy` | 매입평균가 | |
| `price` | 현재가 | API 조회 또는 수동 입력 |
| `manual` | 수동 입력 여부 | true면 새로고침해도 덮어쓰지 않음 |
| `err` | 조회 실패 표시 | |

### 6.2 계산 로직 (검증 완료)

| 계산 | 수식 |
|---|---|
| 매입금액 | 보유수량 × 매입평균가 |
| 평가금액 | 보유수량 × 현재가 |
| 평가손익 | 평가금액 − 매입금액 |
| 수익률(%) | 평가손익 ÷ 매입금액 × 100 |
| 증권사 소계 | 소속 종목들의 매입금액·평가금액 합 |
| 전체 합산 | A + B 증권사 소계의 합 |

> 검증 예시(단위 테스트 통과): 삼성전자 10주 @70,000, 현재 75,000 → +50,000원(+7.14%).
> 카카오 5주 @55,000, 현재 50,000 → −25,000원(−9.09%). 합산 +25,000원(+2.56%).

### 6.3 주요 함수 (수정 시 위치 기준)

| 함수 | 역할 |
|---|---|
| `load()` / `saveLocal()` | localStorage 읽기/쓰기 (오프라인 캐시) |
| `save()` | 로컬 저장 + 클라우드 저장 예약 |
| `render()` | 표·요약카드 전체 다시 그리기 및 손익 재계산 |
| `refreshPrices()` | 종목코드 모아 `/api/price` 호출, 현재가 갱신 |
| `editPrice()` | 현재가 셀 클릭 시 수동 입력 |
| `loadRemote()` | 시작 시 클라우드에서 데이터 불러오기, 접근코드 처리 |
| `scheduleRemoteSave()` / `doRemoteSave()` | 변경 후 0.8초 뒤 클라우드 저장(디바운스) |
| `setSync()` | 상단 "☁ 저장됨 / 저장 중… / 오프라인" 표시 |

### 6.4 색상 규칙

이익·상승은 빨강(`--up`), 손실·하락은 파랑(`--down`)으로 한국 증시 관습을 따른다.
CSS 변수는 `:root`에 모여 있어 한 곳에서 테마 변경이 가능하다.

## 7. API 명세

### 7.1 `GET /api/price?code=NNNNNN` — 현재가 조회

| 항목 | 내용 |
|---|---|
| 데이터 출처 | 한국투자증권 KIS Open API (`inquire-price`, tr_id `FHKST01010100`) |
| 파라미터 | `code` = 6자리 종목코드 |
| 인증 | 서버가 환경변수 키로 토큰 발급(24시간 캐시) |

응답 예시:

```json
{ "code":"005930", "name":null, "price":279000,
  "change":16000, "changeRate":6.08, "updatedAt":"2026-07-15T02:28:58Z" }
```

> 참고: KIS 이 TR은 `name`을 비워 보낼 수 있다(null). 화면은 사용자가 입력한 종목명을 쓰므로 문제 없음.

### 7.2 `GET/POST /api/data` — 포트폴리오 저장/조회

| 메서드 | 동작 |
|---|---|
| `GET` | 저장된 포트폴리오 JSON 반환 (`{ "data": {...} }`) |
| `POST` | 요청 본문(JSON)을 저장 |
| 저장 키 | `portfolio:default` (단일 포트폴리오 공유) |
| 보호 | 환경변수 `APP_PASSCODE` 설정 시 헤더 `x-passcode` 필요 |

> 접근코드가 설정된 경우, 코드 없이 외부에서 `GET /api/data`를 호출하면 401이 반환된다.
> (브라우저는 접근코드를 저장해 두고 자동으로 헤더에 넣는다.)

## 8. 환경변수

Vercel > Settings > Environment Variables 에 등록한다. 코드나 GitHub에는 절대 넣지 않는다.

| 이름 | 필수 | 용도 | 획득처 |
|---|---|---|---|
| `KIS_APP_KEY` | 필수 | KIS 인증 | 한국투자증권 KIS Developers |
| `KIS_APP_SECRET` | 필수 | KIS 인증 | 한국투자증권 KIS Developers |
| `KV_REST_API_URL` | 필수 | 저장소 주소 | Vercel Storage 연결 시 자동 등록 |
| `KV_REST_API_TOKEN` | 필수 | 저장소 인증 | Vercel Storage 연결 시 자동 등록 |
| `APP_PASSCODE` | 선택 | 접근코드 보호 | 직접 지정 |

> 환경변수를 추가·변경한 뒤에는 반드시 **Redeploy** 해야 반영된다.

## 9. 배포 및 운영

### 9.1 배포 파이프라인

GitHub 저장소에 커밋 → Vercel이 자동 감지 → 자동 빌드·배포. 별도 명령 불필요.

| 구분 | 주소 | 특징 |
|---|---|---|
| 정식(Production) | `kospi20k.vercel.app` | 공개 접속. 실제 사용 주소 |
| 미리보기(Preview) | 해시가 붙은 임시 주소 | 배포마다 새로 생성, 로그인 보호 걸림 → 사용 X |

### 9.2 파일 수정 → 반영 절차 (GitHub 웹 기준)

1. GitHub 저장소에서 해당 파일 열기 → 연필(Edit) 또는 Upload files로 교체
2. Commit changes (main 브랜치)
3. Vercel Deployments 탭에서 최신 배포가 "Ready" 될 때까지 대기(1~2분)

### 9.3 KV 저장소 연결(최초 1회)

Storage → Create Database → Upstash for Redis → 프로젝트에 Connect → Redeploy.
연결 시 `KV_REST_API_URL`·`KV_REST_API_TOKEN`이 자동 등록된다.

### 9.4 (대안) 공공데이터포털 방식으로 전환하려면

실시간이 아닌 "전일 종가"로 충분하고 증권사 계좌 없이 운영하려는 경우에만 해당한다.
`api/price.js`를 `getStockPriceInfo`(엔드포인트 `apis.data.go.kr/1160100/service/GetStockSecuritiesInfoService`)
호출로 바꾸고, 환경변수를 `DATA_GO_KR_KEY`(일반 인증키 Decoding)로 교체한다.
응답의 `clpr`(종가)를 현재가로 사용한다. 이 방식은 실시간이 아님에 유의.
